\section{建模目标与范围}
本文档从用例、静态结构、动态交互和状态四个维度对逸仙活动云系统进行建模，目标如下：
\begin{enumerate}
\item 明确系统边界和参与者职责，为功能开发和权限设计提供依据。
\item 定义核心领域实体及其关系，建立前后端对数据模型的共同理解。
\item 通过序列图和状态图展示关键业务流程的动态行为，指导接口设计和测试用例编写。
\end{enumerate}
建模遵循以下约定：模型以"活动"作为外部领域概念，持久化模型中使用 \texttt{Poster}，通过 API 适配层映射为前端的 \texttt{Activity}，图中以"活动"表述业务含义。用例覆盖访客、普通用户、发布者和管理员四种角色，重点展示审核闭环、权限边界及核心数据关系。

\section{用例模型}
\subsection{参与者分析}
系统识别四类参与者：
\begin{enumerate}
\item \textbf{访客：} 未登录用户，主要浏览公开活动信息。是系统的最大用户群体，无需注册即可使用核心查询功能。
\item \textbf{普通用户：} 已注册登录的学生用户，系统默认角色为 \texttt{viewer}。在访客能力基础上可报名、收藏、管理个人日历和接收通知。
\item \textbf{发布者：} 经管理员审批获得发布权限的用户，角色为 \texttt{publisher}。可创建和编辑活动草稿、提交审核并查看审核结果。
\item \textbf{管理员：} 负责平台内容治理和权限管理，角色为 \texttt{admin}。可审核活动、审批发布者申请、管理数据源和查看审计日志。
\end{enumerate}

\subsection{主要属性与操作}
各核心类的关键属性与操作如下：

\begin{enumerate}
\item \textbf{User（用户）：} 属性包括 id、username、email、password\_hash、role（viewer/publisher/admin）。操作包括注册、登录、申请发布者、查看通知。
\item \textbf{Poster（活动）：} 属性包括 id、title、summary、content、event\_time、location、organizer、status（draft/pending\_review/published/rejected）、quality\_score。操作包括创建草稿、编辑、提交审核、审核通过/驳回。
\item \textbf{Registration（报名）：} 属性包括 user\_id、poster\_id、name、student\_id、created\_at。操作包括报名、取消报名、查询报名列表。
\item \textbf{AuditLog（审计日志）：} 属性包括 actor\_id、action、target\_type、target\_id、detail、created\_at。操作包括写入日志、按条件查询。
\end{enumerate}

\subsection{对象交互说明}
\begin{enumerate}
\item \textbf{发布审核流程：} 发布者通过前端提交草稿，API 收到请求后校验 JWT 身份和角色是否为 publisher，验证活动状态是否为 draft。校验通过后将状态更新为 pending\_review 并写入审计日志。管理员获取审核队列，选择通过或驳回，系统更新状态并记录审核意见。非法状态转换（如从 published 再次提交）返回 400 错误。
\item \textbf{报名与幂等流程：} 用户从活动详情页提交报名，前端携带 JWT 发送 POST 请求。API 校验用户身份和活动可报名性后，以 \texttt{(user\_id, poster\_id)} 唯一约束创建报名记录。若同一请求重复到达，数据库唯一约束保证仅一条记录，服务返回既有记录而非创建重复报名。取消报名时根据唯一约束定位并删除对应记录。
\end{enumerate}

\subsection{状态迁移说明}
\begin{enumerate}
\item \textbf{draft $\rightarrow$ pending\_review：} 发布者提交草稿触发，服务层校验必填字段（标题、时间、地点）和角色权限。校验不通过返回 400 和具体错误信息。
\item \textbf{pending\_review $\rightarrow$ published：} 管理员执行批准操作，同时生成审核意见和审计日志，触发知识关联和订阅通知。活动对访客和用户可见。
\item \textbf{pending\_review $\rightarrow$ rejected：} 管理员执行驳回操作，需填写驳回原因。发布者可查看反馈并重新编辑。
\item \textbf{rejected $\rightarrow$ draft：} 发布者在查看驳回原因后修改活动，重新保存为草稿，可再次提交审核。
\item \textbf{draft（编辑/保存）：} 草稿状态允许任意次编辑保存，不触发状态变更。所有字段均为可选，不强制必填校验。
\end{enumerate}
当前实现允许管理员审核待审状态或草稿状态的活动。非法状态转换（如非管理员调用审核接口、非草稿状态提交审核）由服务层拦截并返回错误。生产演化时可补充撤回（published $\rightarrow$ draft）、下线（published$\rightarrow$ archived）和过期（自动 transition）等状态。
