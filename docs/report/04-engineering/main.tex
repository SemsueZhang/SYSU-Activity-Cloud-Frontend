\section{软件工程化目标}
本项目以"每次变更可追踪、可构建、可验证、可交接"为工程化目标，围绕软件过程、协作、自动化和知识沉淀建立轻量但可执行的机制。核心不是堆砌工具，而是让需求—设计—代码—测试—发布形成闭环，并使前后端能在接口演化中保持一致。具体而言，工程化实践旨在达成以下目标：
\begin{enumerate}
\item \textbf{可追踪：} 每个功能从需求提出到上线发布的过程可追溯，变更历史可审计。
\item \textbf{可构建：} 每次提交均可通过自动化构建验证，类型错误和编译问题在合并前被发现。
\item \textbf{可验证：} 构建和测试过程自动化执行，减少人工回归成本，及早发现集成问题。
\item \textbf{可交接：} 环境配置和部署流程可复现，知识以文档形式沉淀，不依赖口头传递。
\end{enumerate}

\subsection{沟通机制}
团队采用以下沟通机制确保信息同步：
\begin{enumerate}
\item \textbf{接口联调：} 前后端在开发新功能前共同确认接口字段、枚举值、分页参数和错误语义，更新契约矩阵。接口变更必须经双方确认并更新文档，仅在聊天中确认而未更新矩阵不得视为完成。
\item \textbf{代码审查：} 每个合并请求至少由一名非作者成员审查，检查内容包括权限边界、异常路径、代码可读性和测试覆盖。审查未通过或 CI 失败的合并请求不得合并。
\item \textbf{变更通知：} 数据库 Schema 变更、接口契约调整或环境配置修改在工作群中同步，确保前后端开发人员及时感知依赖变更。
\end{enumerate}

\section{版本控制规范}
\subsection{Git 仓库管理}
项目分为两个独立仓库，职责清晰分离：
\begin{enumerate}
\item \textbf{前端仓库（SYSU-Activity-Cloud-Frontend）：} 包含 Vue 3 前端源码、端到端测试（\texttt{e2e/}）、Mock 服务（\texttt{mock/}）、CI 工作流（\texttt{.github/workflows/ci.yml}）和设计文档（\texttt{docs/}）。
\item \textbf{后端仓库（Campus-Activity-Information-Platform）：} 包含 Flask 后端源码（\texttt{app/}）、pytest 测试（\texttt{tests/}）、Docker Compose 部署配置（\texttt{docker-compose.yml}）和 CI 工作流（\texttt{.github/workflows/}）。
\end{enumerate}
两个仓库在发布时以兼容版本对组成系统基线，记录前端 commit/tag、后端 commit/tag、接口契约版本和数据库 Schema 版本。主分支始终保持可构建状态，禁止直接推送。

\subsection{分支策略}
实际仓库已按以下分支策略运作：
\begin{enumerate}
\item \textbf{主分支（main）：} 保持可构建、可部署，所有变更通过合并请求引入。
\item \textbf{功能分支（feature/*）：} 从 main 创建，每个分支只解决一个主题。示例：\texttt{feature/auth-register}、\texttt{feature/activity-review}。
\item \textbf{修复分支（fix/*）：} 用于缺陷修复，修复后合并回 main。
\end{enumerate}

\subsection{提交规范}
提交信息遵循清晰描述变更目的的格式。实际仓库中的提交示例：
\begin{verbatim}
feat(auth): 添加邮箱验证码注册接口

- 实现验证码生成与发送接口 POST /api/auth/send-code
- 实现注册接口 POST /api/auth/register 含验证码校验
- 添加对应的 pytest 集成测试
\end{verbatim}
\begin{verbatim}
fix(activity): 修复活动列表分页参数丢失问题

- 修复前端分页组件未传递 page 参数的 bug
- 添加分页参数的 E2E 测试覆盖
\end{verbatim}
提交应小而聚焦，避免将多个无关功能的改动混入同一个提交。禁止提交真实密钥、编译产物（\texttt{dist/}）或本地数据库文件（\texttt{*.db}）。

\subsection{合并请求与代码审查}
每个合并请求需包含以下三项内容：
\begin{enumerate}
\item \textbf{变更说明：} 关联的需求编号（FR-xx）或缺陷编号，以及变更的简要描述。
\item \textbf{验证证据：} CI 运行链接或本地 \texttt{npm run verify} / \texttt{pytest} 的通过截图。
\item \textbf{文档更新：} 接口契约、设计文档或运维文档若有变更需同步更新并附上链接。
\end{enumerate}
审查者的检查清单包括：
\begin{enumerate}
\item 权限边界是否在服务端通过 JWT + RBAC 校验，前端是否隐藏或禁用越权入口。
\item 异常路径（空数据、网络失败、参数非法、越权访问）是否被覆盖。
\item 代码可读性：命名是否清晰、逻辑是否过度嵌套、注释是否必要且准确。
\item 测试覆盖：新增或修改的代码是否有对应的正向和异常测试用例。
\end{enumerate}
CI 未通过、含真实密钥或破坏主干构建的合并请求禁止合并。紧急缺陷可走快速通道，但必须在修复后补齐测试与复盘记录。

\section{开发规范}
\subsection{代码规范}
\begin{enumerate}
\item \textbf{前端：} 使用 TypeScript 严格模式（\texttt{tsconfig.json} 中 \texttt{strict: true}），禁止使用 \texttt{any} 类型；组件采用组合式 API（Composition API）和 \texttt{<script setup>} 语法；样式使用 CSS 类名，避免内联样式；遵循 Vue 官方风格指南。
\item \textbf{后端：} 遵循 PEP 8 编码规范；视图函数（蓝图路由）不包含业务逻辑，业务规则沉淀在服务层（\texttt{services/}）；数据库操作通过 SQLAlchemy ORM，避免裸 SQL。
\item \textbf{通用：} 不允许硬编码敏感信息（密钥、密码、Token）；不允许提交调试代码（\texttt{console.log}、\texttt{print}、断点）；不允许将编译产物和依赖缓存（\texttt{node\_modules/}、\texttt{.venv/}）提交到仓库。
\end{enumerate}

\subsection{命名规范}
\begin{enumerate}
\item \textbf{前端：} 组件文件使用 PascalCase（\texttt{ActivityDetail.vue}、\texttt{LoginView.vue}）；普通 TypeScript 文件使用 camelCase（\texttt{activitySort.ts}、\texttt{activityQuality.ts}）；路由路径使用 kebab-case（\texttt{/activity/:id}、\texttt{/admin/review}）。
\item \textbf{后端：} 蓝图模块使用 snake\_case 文件名（\texttt{auth.py}、\texttt{data\_sources.py}）；API 端点使用小写复数名词（\texttt{/api/activities}、\texttt{/api/auth/login}）；模型类使用 PascalCase 单数形式（\texttt{User}、\texttt{Poster}、\texttt{Registration}）。
\item \textbf{数据库：} 表名使用 snake\_case 复数形式（\texttt{users}、\texttt{posters}、\texttt{registrations}）；字段名使用 snake\_case（\texttt{created\_at}、\texttt{password\_hash}）。
\end{enumerate}

\subsection{文档规范}
项目文档统一使用 Markdown 格式，存放于 \texttt{docs/} 目录。实际仓库中已存在的文档资产包括：
\begin{enumerate}

\item \textbf{接口契约矩阵（\texttt{docs/frontend-plan/BACKEND\_CONTRACT\_MATRIX.md}）：} 表格记录前端字段名、后端字段名、类型、是否必填和示例值。

\item \textbf{开发规范（\texttt{docs/前端开发规范与流程.md}）：} 定义前端编码规范、提交规范和审查流程。
\item \textbf{功能需求文档（\texttt{docs/前端功能需求文档.md}）：} 详细描述各功能模块的页面行为和交互逻辑。
\end{enumerate}
需求变更时同步更新相关的设计文档、接口契约和测试用例，避免文档与代码脱节。

\section{自动化手段}
\subsection{自动化构建}
前端通过 \texttt{npm run build} 触发 \texttt{vue-tsc -b \&\& vite build}，串行执行 TypeScript 类型检查和 Vite 生产构建，输出可部署的静态资源到 \texttt{dist/} 目录。构建错误（类型不匹配、缺失引用）直接终止并输出错误位置，不生成无效构建产物。后端通过 Dockerfile 定义构建过程，安装 Python 依赖并打包 Flask 应用。

\subsection{自动化检查}
\begin{enumerate}
\item \textbf{类型检查：} \texttt{vue-tsc} 在每次构建时对 TypeScript 代码执行严格模式类型检查，在编译阶段捕获接口字段不匹配、空值引用和类型错误。
\item \textbf{代码质量：} 后端 CI 计划集成 pylint 或 flake8 进行静态检查，当前阶段以前端构建时的类型检查和代码审查作为主要质量门禁。
\item \textbf{安全门禁：} 代码审查中对敏感信息泄露、硬编码凭据、缺少输入校验和 SSRF 风险进行人工检查。
\end{enumerate}

\subsection{自动化测试}
三项自动化测试已在项目中落地，集成于 \texttt{npm run verify} 命令：
\begin{enumerate}
\item \textbf{单元测试：} 前端使用 Vitest + \texttt{@vue/test-utils}，覆盖认证 Store（\texttt{src/stores/auth.spec.ts}）、活动排序工具（\texttt{src/utils/activitySort.spec.ts}）、活动质量工具（\texttt{src/utils/activityQuality.spec.ts}）、日历卡片组件（\texttt{src/components/HomeCalendarCard.spec.ts}）和活动详情组件（\texttt{src/views/activity/\_\_tests\_\_/ActivityDetail.spec.ts}）。后端使用 pytest + 内存 SQLite，覆盖认证、活动、审计、数据源、知识、搜索、质量、去重和安全等 20+ 测试文件。
\item \textbf{端到端测试：} 使用 Playwright 驱动 Chromium 浏览器（\texttt{e2e/public-and-auth.spec.ts}），覆盖访客浏览、登录注册、活动搜索等关键用户旅程。
\item \textbf{聚合验证：} \texttt{npm run verify} 串行执行构建、单元测试和端到端测试，作为提交前的完整性检查命令。
\end{enumerate}

\subsection{自动化打包}
前端构建生成静态文件（\texttt{dist/} 目录），可直接部署到 Nginx 或任何 Web 服务器。后端通过 Docker 构建不可变镜像，由 Docker Compose 编排启动。CI 工作流（\texttt{.github/workflows/ci.yml}）在 push 和 PR 时自动触发，在 Ubuntu 环境中执行：
\begin{verbatim}
- uses: actions/checkout@v4
- uses: actions/setup-node@v4
  with:
    node-version: 22
    cache: npm
- run: npm ci
- run: npx playwright install --with-deps chromium
- run: npm run build
- run: npm run test:unit
- run: npm run test:e2e
\end{verbatim}

\section{项目管理工具}
项目使用以下工具支撑工程化实践，每项工具均有对应的仓库证据：
\begin{enumerate}
\item \textbf{GitHub：} 代码托管和版本控制。前端和后端分属两个独立仓库，主分支受保护，所有变更通过合并请求引入。
\item \textbf{GitHub Actions：} 持续集成平台。前端仓库已配置 CI 工作流（\texttt{.github/workflows/ci.yml}），在 push 和 pull request 上自动执行安装、构建、单元测试和端到端测试。工作流定义包含 Node 22 环境配置、npm 缓存和 Chromium 安装步骤。
\item \textbf{Docker Compose：} 后端部署编排工具。\texttt{docker-compose.yml} 定义 API 服务、PostgreSQL、Redis、Celery Worker 和 Celery Beat 的容器化拓扑，单命令（\texttt{docker compose up -d --build}）启动完整后端环境。
\item \textbf{Mock 服务：} 基于 Node.js + Express 的轻量级 API Mock（\texttt{mock/index.js}），实现认证、活动等核心接口的模拟响应，供前端在无后端环境时独立开发和联调。
\item \textbf{Package Scripts：} 前端 \texttt{package.json} 定义了完整的脚本体系——\texttt{dev}（开发服务器）、\texttt{build}（类型检查 + 生产构建）、\texttt{test:unit}（Vitest）、\texttt{test:e2e}（Playwright）、\texttt{verify}（聚合检查）。
\end{enumerate}
