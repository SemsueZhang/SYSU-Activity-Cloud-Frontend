## 1. 技术栈与约定

| 类别      | 选型 / 位置                              | 备注                                                                 |
| --------- | ---------------------------------------- | -------------------------------------------------------------------- |
| 核心框架  | Vue 3 (Composition API, `<script setup>`) | 使用 `<script setup>`，项目使用 TypeScript（`.ts` / `.vue` + TS）      |
| 构建工具  | Vite                                     | 开发服务器端口为 `3000`（见 `vite.config.ts` 中 `server.port`）      |
| UI 组件库 | Element Plus                             | 使用 `element-plus` + `@element-plus/icons-vue`                       |
| HTTP 请求 | Axios                                    | 统一使用 `src/api/client.ts` 中创建的 Axios 实例，`baseURL` 使用 `/api` |
| 路由      | Vue Router 4                             | 路由表定义在 `src/router/index.ts`                                    |
| 状态管理  | Pinia                                    | 全局状态（用户/权限）由 Pinia 管理                                    |
| 语言/类型 | TypeScript                               | 项目为 TS 项目，启用了 `vue-tsc` 类型检查                             |
| 代码风格  | 驼峰命名，组件名 PascalCase               | 组件文件与组合式函数使用清晰命名与注释                                 |

**全局约定**：

- 开发时前端通过 Vite 代理将 `/api` 前缀转发到本地后端（`http://127.0.0.1:5000`），因此 Axios 实例的 `baseURL` 在代码中设为 `'/api'`（见 `src/api/client.ts`）。
- 前端语义名：页面/组件中使用 `Activity`（活动）作为业务术语，API 路径与 Mock 端点一致使用 `/activities`。
- 错误处理由 `src/api/client.ts` 的响应拦截器统一：网络错误提示、HTTP 401/403 特殊处理（登出或跳转登录）、500+ 时提示服务器错误。
- 权限与鉴权：登录后将 `token` 存入 `localStorage`（`token`、`user`），路由守卫在 `src/router/index.ts` 的 `beforeEach` 中检查登录状态并处理权限跳转。
- Mock 服务位于 `frontend/mock/`；本地可启动 `node mock/index.js`（默认监听 5000），或使用 Vite 代理直接连接后端服务。

## 4. API 接口清单（Mock 格式）

> 以下列出 `frontend/mock/` 服务实际注册的端点。启动 Mock 后监听 `http://localhost:5000`，前端通过 Vite 代理 `/api` 前缀转发至此。
>
> 前端 `src/api/activities.ts` 请求路径为 `/activities`，与 Mock 端点 `/api/activities` 对齐。

### 4.1 健康检查

```
GET /api/health
```

无需鉴权。返回示例：

```json
{
  "status": "ok",
  "service": "campus-activity-frontend-mock",
  "timestamp": "2026-05-24T12:00:00.000Z"
}
```

### 4.2 认证模块

#### 登录

```
POST /api/auth/login
Content-Type: application/json
```

请求体：

```json
{
  "username": "admin",
  "password": "admin123456"
}
```

成功响应（200）：

```json
{
  "token": "mock-jwt-token-1716547200000",
  "user": {
    "id": 1,
    "username": "admin",
    "role": "admin",
    "created_at": "2026-01-01T00:00:00"
  }
}
```

失败响应（401）：

```json
{ "message": "invalid credentials" }
```

**预置测试账号**（见 `mock/db.js`）：

| 用户名 | 密码 | 角色 |
|--------|------|------|
| `admin` | `admin123456` | admin（管理员） |
| `zhangsan` | （需查看 db.js 或无密码默认匹配） | publisher（发布者） |

#### 获取当前用户

```
GET /api/auth/me
Authorization: Bearer <token>
```

若 `Authorization` 头携带有效 token，返回 `DEMO_USER`（管理员）：

```json
{ "user": { "id": 1, "username": "admin", "role": "admin", "created_at": "2026-01-01T00:00:00" } }
```

若无 token 或无效：

```json
{ "user": null }
```

#### 发送邮箱验证码

```
POST /api/auth/send-code
Content-Type: application/json
```

请求体：

```json
{ "email": "user@example.com" }
```

成功：

```json
{ "message": "验证码已发送", "code": "123456" }
```

（Mock 模式下验证码直接打印在控制台并返回在响应体中。）

#### 注册（含邮箱验证）

```
POST /api/auth/register
Content-Type: application/json
```

请求体：

```json
{
  "username": "newuser",
  "password": "pass123",
  "email": "user@example.com",
  "verification_code": "123456"
}
```

成功：

```json
{
  "token": "mock-jwt-token-1716547200000",
  "user": {
    "id": 3,
    "username": "newuser",
    "role": "viewer",
    "email": "user@example.com",
    "created_at": "2026-05-24T12:00:00.000Z"
  }
}
```

校验规则（Mock）：
- `username` 长度 2–50
- `password` 最短 6 位
- `email` 需含 `@`
- `verification_code` 必须与发送的验证码一致

## 5. 权限对应表

角色定义来源于 Mock 数据及 Pinia auth store（`src/stores/auth.ts`）。

| 角色 | 描述 | 前端判断 | 可见操作 |
|------|------|----------|----------|
| `admin` | 管理员 | `isAdmin`（`user.role === 'admin'`） | 审核活动、系统管理、查看所有内容；登录后侧边栏显示管理入口 |
| `publisher` | 发布者 | `isPublisher`（`user.role === 'publisher'`） | 创建/管理活动；可以访问发布相关功能 |
| `viewer` | 普通用户 | 已登录但 role 非 admin/publisher | 浏览、评论、收藏、查看个人中心（待实现） |
| guest | 未登录 | `!isLoggedIn` | 仅浏览首页与公开内容；点击需要登录的功能时跳转登录页 |

**当前实现情况**：
- 路由守卫区分已登录/未登录（未登录访问受限路由跳转 `/auth/login`）。
- 组件内可使用 `auth.isAdmin` / `auth.isPublisher` 控制按钮/区域显示。
- 更细粒度的权限判断（如“仅创建者可编辑”）待后续实现。

## 6. 关键交互流程说明

### 6.1 登录流程

1. 用户在 `LoginView` 输入用户名+密码，点击提交。
2. `src/stores/auth.ts` 中的 `login()` 调用 `POST /api/auth/login`。
3. Mock 验证用户名密码（预置：`admin/admin123456`），返回 token 与用户信息。
4. Store 将 `token` 和 `user` 写入 `localStorage`，并设置 Axios 请求头 `Authorization: Bearer <token>`。
5. 路由守卫检测到已登录，重定向至首页 `/`。

### 6.2 注册流程

1. 用户在 `RegisterView` 输入邮箱，点击“发送验证码”。
2. 调用 `POST /api/auth/send-code`，Mock 控制台打印验证码并返回在响应中。
3. 用户输入验证码、用户名、密码，提交注册。
4. 调用 `POST /api/auth/register`，校验通过后返回 token 和 user（role 为 `viewer`）。
5. 自动登录并跳转首页。

### 6.3 首页加载流程

1. 进入 `/`，`HomeView` 挂载后 `useHomePage` 的 `fetchData()` 执行。
2. 并发调用两次 `listActivities()`：
   - 一次筛选 `status=published` → 热点活动（`hotActivities`，用于轮播）。
   - 一次获取全部活动 → 最新活动（`recentActivities`，用于分类列表）。
3. 若已登录，同时调用 `getCalendarEvents()` 获取当日日程。
4. 热点轮播自动 4 秒切换；分类 Tab 切换时前端过滤 `activity_type`。

### 6.4 退出登录

1. 点击 NavSidebar 或 AppHeader 中的退出按钮。
2. `auth.logout()` 清除 `localStorage` 中的 `token` 和 `user`，重置 Pinia 状态。
3. 路由跳转至首页 `/`，页面变为未登录状态。

### 6.5 查看活动详情

1. 在活动列表或轮播中点击某活动卡片。
2. 路由跳转至 `/activity/:id`，组件加载时调用 `GET /api/activities/:id` 获取详情数据。
3. 展示活动完整信息。
4. 报名、分享、收藏按钮当前统一提示“内容未实现！”，用于明确标识未接入的交互。
