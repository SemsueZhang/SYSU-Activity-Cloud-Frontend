# 校园活动信息平台：全站建设工作清单

> 版本：1.0  
> 日期：2026-07-13  
> 用途：将现有前端需求、页面设计与当前代码状态拆解为可执行、可验收的全站建设任务。  
> 工作方式：所有新增页面或明显 UI 变更，必须先补充 `docs/frontend-design/` 中的设计文档，再开始实现。

> **交付状态更新（2026-07-14）**：本清单中的未勾选项保留为原始规划记录，不能代表当前代码状态。公开端、认证、个人中心、日历、发布者工作区、管理员审核/数据源/审计均已实现并通过 Mock 下的构建、单测与 E2E 验证。真实后端未冻结的报名、独立收藏、资料更新和密码重置仍需按 [契约矩阵](BACKEND_CONTRACT_MATRIX.md) 与 [联调清单](BACKEND_INTEGRATION_CHECKLIST.md) 确认；这不是前端交付阻塞，而是联合测试前置条件。

## 1. 依据、范围与使用方式

### 1.1 依据

- [前端功能需求文档](../前端功能需求文档.md)：技术约定、已实现页面、角色定义和后续页面方向。
- [前端开发规范与流程](../前端开发规范与流程.md)：设计先行、按实际使用创建 API/Mock、只注册已实现路由。
- `docs/frontend-design/`：首页、登录、注册、活动详情的现有视觉与交互基线。
- 当前代码：以 `src/router/index.ts`、`src/api/`、`mock/` 的实际状态作为完成状态依据。

### 1.2 全站范围

本清单覆盖公开活动发现、账户与个人空间、活动发布、管理员审核与系统管理，以及质量保障和发布。后端仓库不在当前工作区；所有尚未冻结的真实接口均应先完成契约确认，不能凭前端猜测字段或路径。

## 2. 全局设计与工程约束（先完成）

### 2.1 需求与接口基线（P0）

- [ ] 整理“前端路由—页面—角色—后端接口—Mock”矩阵，存放于本目录或需求文档附录。
- [ ] 与后端确认活动实体的最终命名和字段映射：当前前端使用 `Activity`、`/activities`，后端说明中存在 `poster` 相关术语；统一后再扩展 API 层。
- [ ] 确认以下接口是否已可用及其错误码、分页格式、权限规则：搜索、报名、收藏、订阅、通知、日历、活动 CRUD、审核、数据源、字典、审计日志、导出。
- [ ] 为每个待建页面创建设计文档，至少说明：目标用户、入口/返回路径、布局、加载/空/错误/无权限状态、接口合同、Mock、响应式与验收条件。
- [ ] 清理或标注过期文档：`前端本地测试指南.md` 中的旧路由和 `mock-server.js` 引用应与实际 `mock/index.js`、`/auth/*`、`/activity/:id` 对齐。

**验收**：团队可从矩阵中明确任何页面的入口、权限、数据来源和未决接口；不存在把未实现入口注册成 404 的情况。

### 2.2 视觉系统与应用外壳（P0）

- [ ] 固化现有视觉 token：品牌深绿 `#0d5e3c`、辅助绿 `#27a66b`、浅绿背景、白色圆角卡片、柔和阴影、圆角按钮和中文优先排版。
- [ ] 在 `src/style.css` 建立颜色、间距、圆角、阴影、字号、断点 CSS 变量；仅抽取稳定的公共样式，不提前抽象业务组件。
- [ ] 定义公共页面框架：公开页顶栏/页脚、登录后工作区顶栏/侧栏、内容最大宽度、面包屑、返回行为。
- [ ] 统一页面状态组件/样式：加载骨架、空数据、请求失败、404、403、无搜索结果、提交中和成功反馈。
- [ ] 定义并实施桌面、平板、手机断点；任何新页面至少验证 1440px、1024px、768px、375px 宽度。
- [ ] 为可点击的卡片、图标按钮、输入框、弹窗与颜色状态制定键盘焦点、`aria-label`、语义标签和对比度标准。

**验收**：新页面无需复制粘贴基础配色或状态样式；在窄屏下无水平溢出，键盘可依次访问所有核心操作。

### 2.3 前端基础能力（P0）

- [ ] 将路由守卫扩展为明确的 `requiresAuth`、`roles` 元信息；未登录保留 `redirect`，无角色权限跳转至明确的 403/首页提示。
- [ ] 审核 token 过期行为：清理 Store 与 `localStorage`、只提示一次、回到带 `redirect` 的登录页。
- [ ] 补齐鉴权初始化策略：页面刷新时用 `GET /auth/me` 校验 token，并处理用户资料变化。
- [ ] 统一 API 类型：分页响应、标准错误、时间字段、活动状态、角色枚举，避免页面自行定义重复结构。
- [ ] 增加请求取消或竞态保护，避免搜索输入、筛选切换、路由快速跳转时旧响应覆盖新数据。
- [ ] 建立可复制的 Mock 路由模板，支持正常、空、401/403、404、422、500 与慢响应场景。

**验收**：受保护路由、角色路由和过期 token 的跳转可由单测验证；新增接口有类型、成功/失败行为和 Mock 场景。

## 3. 公开活动发现（P0）

### 3.1 首页收尾（`/`）

- [ ] 更新首页设计文档，使其与当前实际交互一致（热门卡片现已进入详情，不再仅提示未实现）。
- [ ] 让“全部活动”“热门推荐”“按类别”导航有可见、可访问的定位目标；没有对应区块时不显示误导入口。
- [ ] 完成搜索提交：携带 `q` 跳转 `/search?q=...`，保留输入关键词并支持 Enter 提交。
- [ ] 完成“查看全部”活动入口至 `/activities` 或 `/search`（以最终信息架构为准）。
- [ ] 使日历选中日期能筛选/提示当天活动，并明确个人日程和公开活动的展示边界。
- [ ] 为热点轮播补充暂停/恢复、键盘可操作性、减少动态效果偏好支持，以及空数据、单条数据、网络失败验证。
- [ ] 按接口能力决定分类过滤是前端已加载数据过滤还是服务端分页筛选；数据量增长时切换为服务端筛选。

**验收**：访客可从首页搜索、按分类浏览、打开活动详情；无活动、慢请求和接口失败均有稳定且不跳动的显示。

### 3.2 搜索结果页（`/search`，P0）

**先建设计文档：** `docs/frontend-design/SEARCH_DESIGN.md`。

- [ ] 定义 URL 状态：`q`、搜索范围（内部/外部）、页码、排序、类别及可选地点/时间筛选。
- [ ] 实现初始搜索框、内部/外部搜索切换、搜索建议的加载与防抖（若后端提供建议接口）。
- [ ] 实现结果卡片：标题、摘要、高亮关键词、类型、时间、地点、主办方、来源/匹配度（仅在接口提供时展示）。
- [ ] 实现筛选、排序、分页；筛选操作同步 URL，刷新和分享链接可恢复状态。
- [ ] 实现空关键词、零结果、加载中、请求失败与外部搜索不可用的状态。
- [ ] 搜索结果活动可进入 `/activity/:id`；外部结果有安全的外链标识和 `rel="noreferrer"`。
- [ ] 新增 `src/api/search.ts` 和最小 Mock 路由；不要复用活动列表接口去伪造外部搜索。

**验收**：通过 URL 直接打开搜索结果能复现筛选和分页；内部与外部搜索失败不会影响彼此的可用性。

### 3.3 公共活动列表（`/activities`，P0）

- [ ] 决定该页面是公开发现页还是登录后的管理入口；若两者需求不同，分别使用 `/activities` 和 `/my/activities`，不混合权限逻辑。
- [ ] 实现卡片/列表切换（仅在设计确认需要时）、关键词、分类、日期、地点、状态（管理态）筛选和分页。
- [ ] 复用已确定的活动摘要展示，不复制首页卡片的业务逻辑。
- [ ] 提供排序：最新发布、活动时间、相关度（搜索来源）；缺少后端支持时不暴露不可用排序项。

**验收**：列表能在大数据分页下稳定工作，并且所有筛选值有明确可清除的 UI 与 URL/状态归属。

### 3.4 活动详情完成闭环（`/activity/:id`，P0）

- [ ] 更新 `ACTIVITY_DETAIL_DESIGN.md`，明确当前的公开可见规则、状态可见性和真实接口字段。
- [ ] 对无效 ID 显示独立 404 内容和返回首页操作，不能只弹消息后静默跳转。
- [ ] 实现报名：未登录跳转 `/auth/login?redirect=<当前完整路径>`；已登录调用报名接口、禁用重复提交并更新报名人数。
- [ ] 实现收藏/取消收藏与收藏状态；接口未具备前保持明确的未实现提示，不伪造成功状态。
- [ ] 实现分享：优先 Web Share API，降级为复制当前链接；复制结果有可感知反馈。
- [ ] 将正文按数据类型安全展示：纯文本保留换行；若后端返回 HTML，先约定并接入可信的净化策略，禁止直接 `v-html` 渲染不可信内容。
- [ ] 附件增加文件类型/大小（接口提供时）、打开/下载行为与缺失资源容错；图片增加替代文本、懒加载与预览（设计确认后）。
- [ ] 增加相关活动区块；仅在后端提供关联接口后接入，不用随机活动代替。

**验收**：游客、已登录用户、重复报名、404、加载失败均有可验证的预期行为；详情内容不引入 XSS 风险。

### 3.5 公开信息页（P2）

- [ ] 建立 `/about` 设计与页面：平台定位、数据来源、内容审核边界、隐私/免责声明入口。
- [ ] 建立 `/contact` 设计与页面：反馈渠道、问题分类、联系信息；若有反馈接口再加表单与频率限制提示。
- [ ] 在页脚只放已实现且已审核的外部链接，所有外链标识打开方式。

**验收**：页面内容有明确来源和更新时间；不包含虚构联系方式或不可用表单。

## 6. 管理后台（P1）

### 6.1 管理入口与看板（`/admin`）

- [ ] 先建设计文档，定义管理菜单、统计卡片和小屏导航方式。
- [ ] 为 `admin` 路由添加角色守卫；非管理员展示明确的无权限结果。
- [ ] 实现服务端可提供的概览指标：待审数量、已发布量、数据源状态、通知/任务异常；没有对应接口时不放假指标。
- [ ] 从看板进入审核、数据源、字典、审计和导出页面。

**验收**：管理员能在一个入口到达所有已实现的后台能力；普通用户不能读取管理数据。

### 6.2 审核队列（`/admin/review`）

- [ ] 先建设计文档：筛选条件、列表列、单条详情抽屉/页面、批准/驳回、批量操作和失败反馈。
- [ ] 实现待审列表、状态/来源/重复分组筛选、质量分排序和分页。
- [ ] 实现批准、驳回理由、批量操作、二次确认和部分失败结果呈现。
- [ ] 审核完成后同步列表、活动详情和统计数据；记录由后端完成，前端展示返回结果。

**验收**：批量操作中的单项失败不会掩盖成功/失败明细；所有敏感操作需要确认且记录有反馈。

### 6.3 数据源、字典与图谱管理（P1/P2）

- [ ] 数据源（P1）：列表、创建/编辑、启停、手动抓取、任务状态/日志入口；危险动作需确认。
- [ ] 受控词表（P2）：地点、组织、主题的别名管理，校验重复与影响范围。
- [ ] 知识图谱（P2）：节点列表、类型筛选、节点详情、关联活动；图谱可视化仅在数据和交互设计成熟后引入。
- [ ] 对长时间任务使用轮询或服务端推送，并提供离开页面后的正确取消/停止策略。
